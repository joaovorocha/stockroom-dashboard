Project Overview 

Nexgen  Packaging  will  provide  pressure  sensitive  RFID  labels  and  non-RFID  barcode  labels  from 
Service Bureau out of the UK, China (Shenzhen) and Hong Kong production sites. 

●  Summary of the Project 

○  Existing customer that is changing their current XML data feed to a JSON Data feed 
■  Will require HK IT support to ingest new JSON format directly through Boomi 
into the existing Suitsupply Custom DB Structures (ie. Not edi-automation) 

○  Currently  Suitsupply  only  has  RFID  items,  but  will  now  be  adding  new  NON-RFID 

items 
●  Summary of the Data files 

○  Suitsupply  is  moving  away  from  their  current  XML  data  files.  Instead,  they  are 
combining both RFID and NON-RFID Purchases orders into a JSON format (not edi 
automation format) 

○  There will be 3 distinct files: Products, Purchase Orders, Transfers 

■ 

■ 

■ 

“Products”  file:  will  be  Product  Master  data  that  will  need  to  be  stored 
separately 
“Purchase Orders” file: will be PO data that will need to be joined with the 
Products data downstream 
“Transfers” file: will be similar/identical to Purchase Orders file 

●  UAT Test Scripts 

○  SuitSupply UAT Test Scripts gsheet.

Program Details 

1.  Nexgen to start production in the UK and Hong Kong simultaneously. 
2.  TID and EPC to be captured, stored in the cloud, and published back to Suitsupply in a 
weekly print report with additional data elements captured and encoded at print stage. 
3.  Suitsupply and vendors to use Nexgen Ecom system to call out orders via the standard 

Purchase Order call out process whereby Suitsupply will provide electronic JSON data files 
via web service in batch daily and Nexgen will publish to the vendor/factory users to 
execute in which all variable data will be provided direct from Suitsupply IT.  

Packing Instructions 

1.  Labels must be packed in their own poly bag – separated out by PO.  A dot will indicate the 

start of a new PO: 

2.  Packing lists should be included in each package as per normal. 
3.  Larger order quantities of 250 labels+ should be wound onto rolls. 
4.  Smaller quantities can simply be folded and packed neatly into a poly bag 
5.  All Separate packages from the same order should then be combined and packed together 
with paper or bubble wrap to protect the labels from damage during transit when shipping. 

6.  Multiple orders should be shipped together.

Workflow 

- 

Suitsupply will send the ‘Product’ data files  

o  Nexgen will store the product data to be joined with the ‘Purchase Order’ data later 
o 

In case Suitsupply sends updated ‘Product’ details for an existing ‘SkuCode’, 
Nexgen will update the ‘Product’ database overwriting the old data. 

- 

Suitsupply will send ‘Purchase order’ data file  

o  Nexgen will go find the data associated with each ‘SkuCode’ in the ‘Product’ 

database and retrieve the data to print the labels  

▪ 

‘Quantity’ field in the PO data file indicates the amount of that SKU 
ordered. 

▪  Nexgen will check the type of labels and how many of them are required 

according to the product category. See point 2 of the ‘Product 
Specifications’ section of this document. 

●  Example: if ‘Quantity’ is 2 for a suit which requires 2 labels, 

Nexgen will print 4 large labels. 

- 

Suitsupply will send a ‘Transfer’ data file 

o  Nexgen will go find the data associated with each ‘SkuCode’ in the ‘Product’ 

database and retrieve the data to print the labels  

‘Quantity’ will be filed in the ODP interface. 

▪

point 2 of the ‘Product Specifications’ section of this document.

API for JSON 
Suit Supply currently utilizes an API webservice to pass an XML payload which is stored 
as an XML file in their Nexgen FTP Folder. Pulse is designing a new API that can accept a 
json payload, which will then be stored as a JSON file in the same Nexgen FTP Folder.
